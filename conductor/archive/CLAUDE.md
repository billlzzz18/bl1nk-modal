## Repository layout

This is a **monorepo of independently-deployed Modal apps** — there is no shared root `pyproject.toml`, workspace tool, or build system. Each app directory is a self-contained Python (or Rust) project with its own dependencies and its own test suite.

```
modal-apps/
  modal-runner/     # FastAPI webhook → CrateDB task tracker + LINE Notify, deployed on Modal
  modal-agy/        # Orchestrator for the `agy` (Antigravity) agent sandbox
  modal-sandbox/    # General-purpose sandbox API (create/exec/upload/download via modal.Sandbox)
  modal-opencode/
    engine/         # `sovereign_engine` — Rust/pyo3 crate for issue/PR label auto-detection
modal-images/       # Shared dev image build (Rust/Node/Bun/gh/Claude CLI) + bl1nk-search (embedding/reranker service)
conductor/          # Project context generated by the conductor plugin — read this first (see above)
```

There is no root-level Python package — never assume `modal-apps/` or `modal-images/` can be imported as a package from the repo root.

## Commands

Every Python project is independent. `cd` into it before running anything.

**Use `uv sync`, not `pip install -e ".[dev]"`.** Every project declares its dev dependencies under `[dependency-groups]` (PEP 735), which the `.[dev]` extras syntax does not pick up — pip will silently skip them (or, without the `[tool.hatch.build.targets.wheel] bypass-selection = true` setting these hatchling-backed projects also carry, fail outright since there's no conventional package layout to build a wheel from). `uv sync` reads `[dependency-groups]` correctly and creates a project-local `.venv`.

### modal-runner

```bash
cd modal-apps/modal-runner
uv sync
uv run pytest                          # full suite (parse/signature/notify/db/webhook tests)
uv run pytest tests/test_webhook.py    # single file
modal serve main.py                    # local dev server
modal deploy main.py                   # deploy (registers ASGI app + cron jobs)
```

### modal-agy

```bash
cd modal-apps/modal-agy
uv sync
uv run pytest
```

### modal-sandbox

```bash
cd modal-apps/modal-sandbox
uv sync
uv run ruff check .             # lint
uv run pytest                   # tests
modal deploy modal_app.py --name modal-sandbox-v2.1   # per scripts/publish.sh
modal shell modal_app.py::dev   # interactive shell in the dev container (scripts/shell.sh)
```

### modal-images

```bash
cd modal-images
uv sync
uv run pytest
modal run build_bl1nk_rust.py      # build+publish the shared bl1nk-rust image
modal run build_bl1nk_search.py    # build+publish the bl1nk-search image
```

### modal-opencode/engine (Rust)

```bash
cd modal-apps/modal-opencode/engine
cargo fmt --check
cargo test                      # or: cargo test <name> to filter
cargo clippy --all-targets
```

### Repo-wide

```bash
pip install pre-commit
pre-commit install
pre-commit run --all-files      # runs generic hygiene hooks + cargo fmt/test (scoped to the Rust engine)
```

There is **no `package.json` anywhere in this repo** — don't reach for npm/Husky/lint-staged tooling; pre-commit hooks go through the Python `pre-commit` framework in `.pre-commit-config.yaml`.

## Environment setup (first install / a new OS)

Every project that needs secrets ships a `.env.example` next to its `pyproject.toml` — copy it to `.env` (gitignored) and fill in values, then load it per-command with `uv` (no `python-dotenv` dependency needed):

```bash
uv run --env-file .env pytest
uv run --env-file .env modal serve main.py
```

| File | Covers |
| --- | --- |
| `.env.example` (repo root) | `MODAL_TOKEN_ID`/`MODAL_TOKEN_SECRET` — non-interactive Modal auth, for headless machines/Termux where `modal setup`'s browser OAuth flow doesn't work |
| `modal-apps/modal-runner/.env.example` | CrateDB connection, LINE Notify, webhook signature secrets |
| `modal-images/.env.example` | `bl1nk-search` bearer token auth |
| `modal-apps/modal-opencode/.env.example` | GitHub App credentials for the `opencode.py` webhook gateway |

In production these are Modal secrets (`modal secret create ...`), not env vars — the `.env.example` files are for running things locally only. `modal-agy` and `modal-sandbox` don't need one; neither reads any app-level env vars.

## Formatting, line endings, and Windows setup

Root-level `.editorconfig` and `.gitattributes` keep formatting and line endings consistent across every directory in this monorepo (every editor/IDE picks up `.editorconfig` automatically; Git normalizes checked-in files to LF per `.gitattributes`, `uv.lock`/`Cargo.lock` are marked generated so they don't pollute diffs).

**Python formatting:** every Python project (`modal-runner`, `modal-agy`, `modal-sandbox`, `modal-images`) declares `ruff` in `[dependency-groups].dev` with a matching `[tool.ruff]` block (`line-length = 100`, `target-version` pinned to that project's `requires-python`) and has a scoped `ruff-<name>` hook in `.pre-commit-config.yaml`. Run it per project: `cd <project> && uv run ruff check . && uv run ruff format .`

**Whitespace/line-ending sweep:** `.pre-commit-config.yaml`'s `trailing-whitespace`/`end-of-file-fixer` hooks only touch staged files at commit time. To normalize the whole tree in one pass (e.g. after a large merge, or to check CI-side), use the root-level script instead of reaching for `sed`/sed-per-OS scripts — it's pure-stdlib Python so it runs the same on Linux/macOS/Windows/Termux:

```bash
python3 scripts/fix_whitespace.py          # fix in place
python3 scripts/fix_whitespace.py --check  # report only, exit 1 if dirty (CI)
```

**Windows bootstrap:** `scripts/install-windows.ps1` installs everything needed to work on this repo from a clean Windows machine — `uv`, the Rust toolchain (rustup, for `modal-apps/modal-opencode/engine`), the Modal CLI, and `pre-commit` — then runs `uv sync` in every Python project directory and installs the repo's pre-commit hooks. Idempotent; re-running skips anything already installed.

```powershell
powershell -ExecutionPolicy Bypass -File scripts\install-windows.ps1
```

## Architecture notes that span multiple files

### Apps are isolated; nothing shares runtime state

Each `modal-apps/*` directory (plus `modal-images/`) is deployed as its own independent Modal `App`. They share no database, no in-process state, and no import graph — the only thing they share is the base `bl1nk-rust` image (built once in `modal-images/build_bl1nk_rust.py`, referenced via `modal.Image.from_name("bl1nk-rust:latest")` everywhere else) and the general Modal API surface conventions documented in `.claude/skills/modal-image-builds/` and `.claude/skills/modal-image-dev/`.

Every consumer pins `:latest`, so a new image publish reaches all of them automatically — no per-app update needed. **What used to need manual, error-prone updating was the build scripts themselves**: `build_bl1nk_rust.py` didn't actually publish any tags at all (despite docs claiming it did), and `build_bl1nk_search.py` hand-typed the version and today's date into 3 separate `built.publish(...)` calls. Both now call `modal-images/_tags.py`'s `publish_versioned(built, name, major)`, which computes the dated tag and publishes all three. Bumping a major version is a one-line edit to the `MAJOR_VERSION` constant at the top of the relevant build script — never hand-edit a `publish(...)` call or duplicate a version string.

**Build/deploy split convention** (see `.claude/skills/modal-image-builds/references/modal-1.5.1-notes.md`): build scripts use `modal.App.lookup(name, create_if_missing=True)`; deploy scripts use bare `modal.App(name)`. This is intentional, not a bug — don't "fix" one to match the other. Also: Modal 1.5.1's `@app.function()` has **no `mounts=` parameter** — passing one raises `TypeError`; use `image.add_local_file(..., copy=True)` at build time instead.

### Flat-script Python projects need `bypass-selection` for hatchling

`modal-runner`, `modal-agy`, and `modal-sandbox` are single-file apps (`main.py` / `modal_app.py`), not installable packages — there's no `src/<name>/` layout for hatchling to auto-detect. Each carries `[tool.hatch.build.targets.wheel] bypass-selection = true` in its `pyproject.toml` so `uv sync` (and any editable install) doesn't fail with hatchling's "unable to determine which files to ship" error. Don't remove this when editing `pyproject.toml`. `modal-images` avoids the issue by using the `setuptools` backend with explicit `py-modules` instead.

### Two identically-named `modal_app.py` files

`modal-apps/modal-agy/modal_app.py` and `modal-apps/modal-sandbox/modal_app.py` are two different modules with the same filename. Their `tests/conftest.py` files load them under unique names (`modal_agy_app`, `modal_sandbox_app`) via `importlib.util.spec_from_file_location` instead of a plain `sys.path` insert + `import modal_app`, specifically so both test suites can be collected together in one `pytest` run without one shadowing the other in `sys.modules`. If you add a third app with a `modal_app.py`, follow the same pattern.

### `sovereign_engine` label-detection pipeline

`modal-apps/modal-opencode/engine/src/lib.rs`'s `resolve_full_state()` is the entry point pyo3 exposes to Python. The pipeline is: `Detector` (regex on title/body text) + `FileDetector` (path/extension based) → merged into a label set by `Resolver` (handles manual add/remove and mutually-exclusive groups like `stage:`/`type:`/`p:`) → `PolicyEngine::enforce()` (priority escalation on `blocking`, a guardrail that pulls `stage:finalize` back to `stage:review` if code changed without `rev:ready`, and forced `type:dep` on dependency-file changes) → `Resolver::ensure_defaults()` fills in missing `stage:`/`size:`/`p:`. Each stage lives in its own file with its own `#[cfg(test)] mod tests` — read `resolver.rs` and `policy.rs` together, since `resolve()` calls `PolicyEngine::enforce()` internally.

The label taxonomy this produces (`type:`, `stage:`, `size:`, `lang:`, `env:`, `plat:`, `rev:`, `constraint:`, `agent:` prefixes) is **unrelated** to the plain `bug`/`enhancement`/`needs-triage`/`ready-for-agent`/etc. labels used by `.claude/skills/triage` — don't conflate the two systems.

There is a GitHub Actions workflow for this engine's label automation at `modal-apps/modal-opencode/.github/workflows/bl1nk-labels.yml`, but **GitHub Actions only discovers workflows in `.github/workflows/` at the repo root** — this one is nested inside `modal-apps/modal-opencode/` and currently never runs. Don't assume it's active.

### bl1nk-search: spec vs. implementation

`BL1NK_SEARCH_V1_SPEC.md` at the repo root is the target API contract for the search service (soft-delete by default, cursor pagination, versioned updates, R2-backed storage, `trace_id` on every response). `modal-images/search_service.py` is the current implementation and does **not** yet match the spec in several ways: deletes are hard (no soft-delete), there's no cursor pagination, and storage is in-memory only (FAISS `IndexFlatIP` + a plain dict, no persistence, no R2). `conductor/tech-stack.md`'s storage-adapter decision log describes the planned fix (vector store / DB / storage adapters). Don't assume the spec describes current behavior — check the code.

### Test isolation for heavy ML dependencies

`modal-images/tests/conftest.py` stubs `torch`, `transformers`, and `faiss` with minimal fake modules (a real `numpy`-backed brute-force `IndexFlatIP`) so the `bl1nk-search` test suite runs without installing multi-GB packages. If you add tests that need real tensor behavior, extend the stub rather than pulling in the real dependency.

## Cross-platform constraint

Client-side tooling (the Rust CLI/desktop app planned in `conductor/tech-stack.md`, and any shell scripts) must run on every OS the repo owner works from, **including Termux on Android**. `modal-apps/modal-sandbox/scripts/build.sh` used to hardcode a Termux clone path (`/data/data/com.termux/files/home/modal/...`); it's now fixed to resolve `modal-images/` and its own `modal_app.py` relative to the script's own location (`SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"`), the same pattern already used in `modal-apps/modal-agy/run.sh`. **Follow that pattern in any new script** — never hardcode an absolute clone path or assume a standard Linux/macOS environment.
