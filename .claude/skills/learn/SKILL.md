# /learn — ดึงรูปแบบการทำงานที่นำกลับมาใช้ใหม่ได้

วิเคราะห์บทสนทนาปัจจุบัน ดึงรูปแบบการทำงานที่มีคุณค่าและนำกลับมาใช้ซ้ำได้
บันทึกเป็นไฟล์จริงที่ Claude Code โหลดใช้งานได้ — ไม่ใช่แค่บันทึกเก็บไว้เฉยๆ

**หมายเหตุสำคัญ:** ไฟล์นี้เป็นสกิล (`.claude/skills/learn/SKILL.md`) ไม่ใช่คำสั่ง
(`commands/learn.md`) เพราะพฤติกรรม "เสนอแนะให้รัน `/learn` เองโดยอัตโนมัติ"
ต้องใช้กลไกของสกิล — คำอธิบาย (description) ของสกิลจะถูกประเมินทุกเทิร์นของ
บทสนทนาเพื่อตัดสินใจว่าควรเสนอหรือไม่ ส่วนไฟล์ในโฟลเดอร์ `commands/` จะไม่ถูก
ประเมินแบบอัตโนมัติ — ทำงานเฉพาะตอนผู้ใช้พิมพ์ `/ชื่อคำสั่ง` เท่านั้น พิมพ์
`/learn` เพื่อเรียกสกิลนี้ได้เหมือนคำสั่งปกติ ไม่มีอะไรเปลี่ยนจากมุมผู้ใช้

## สิ่งที่ควรนำมาบันทึก

### 1. รูปแบบการแก้ไขข้อผิดพลาด
- ข้อผิดพลาดคืออะไร สาเหตุที่แท้จริงคืออะไร วิธีแก้ที่ได้ผลคืออะไร
- นำไปใช้ซ้ำกับข้อผิดพลาดลักษณะเดียวกันได้หรือไม่

### 2. เทคนิคการค้นหาสาเหตุปัญหา
- ขั้นตอนตรวจสอบปัญหาที่ไม่ชัดเจนในทันที
- การใช้เครื่องมือหลายอย่างร่วมกันจนได้ผลลัพธ์ที่เป็นประโยชน์

### 3. วิธีแก้ไขทางเลือกเฉพาะบริบท
- ข้อจำกัดหรือพฤติกรรมเฉพาะของไลบรารี/บริการภายนอก/เวอร์ชันใดเวอร์ชันหนึ่ง

### 4. รูปแบบการทำงานที่สอดคล้องกับโปรเจกต์นี้
- กฎการเขียนโค้ด หลักการออกแบบที่ตัดสินใจแล้ว รูปแบบการเชื่อมต่อระหว่างส่วนต่างๆ
- **หมวดนี้มักเป็นความรู้เฉพาะโปรเจกต์** — ดูหัวข้อ "ขอบเขต: โปรเจกต์ vs. ทั่วเครื่อง" ด้านล่าง

## เกณฑ์การเสนอแนะให้รัน /learn (ใช้ระหว่างบทสนทนาปกติ ไม่ใช่ตอนกำลังรัน /learn)

ประเมินทุกเทิร์น ไม่ใช่แค่ตอนคุยเรื่องนี้โดยตรง:

1. **งานกิจวัตรที่ยังไม่เคยบันทึกเป็นทางการ** — ทำตามขั้นตอนตายตัวได้ แต่ยังไม่มี
   สกิล/คำสั่งรองรับ แม้ผู้ใช้จะรู้ขั้นตอนอยู่แล้วในใจก็ตาม
2. **ข้อผิดพลาดที่เกิดจากพฤติกรรมเป็นนิสัย** — พลาดซ้ำๆ, ใช้ผิดวัตถุประสงค์เป็นประจำ,
   หรือ workaround ชั่วคราวที่กลายเป็นมาตรฐานโดยไม่มีใครบันทึกไว้
3. **กระบวนการที่มีขั้นตอนชัดเจนและทำซ้ำได้** — ประเมินความถี่และประโยชน์ก่อนเสนอ
   ไม่ใช่ทุกกระบวนการที่ทำซ้ำได้จะคุ้มค่าบันทึก

อย่าเสนอ `/learn` สำหรับ:
- การพิมพ์ผิด/ข้อผิดพลาดทางไวยากรณ์ครั้งเดียว (เว้นแต่เกิดซ้ำจนเข้าเกณฑ์ข้อ 2)
- ปัญหาที่เกิดครั้งเดียวไม่มีรูปแบบ เช่น บริการภายนอกล่มชั่วคราว

## ขอบเขต: โปรเจกต์ vs. ทั่วเครื่อง

**ต้องถามผู้ใช้ทุกครั้งก่อนบันทึก** ว่าสิ่งที่เจอเป็นความรู้เฉพาะโปรเจกต์นี้
(เช่น สถาปัตยกรรม, ข้อตกลงในโค้ดเบสนี้, บั๊กเฉพาะไฟล์นี้) หรือเป็นนิสัย/ความรู้
ส่วนตัวที่ใช้ได้ทุกโปรเจกต์ (เช่น "ฉันมักลืมตรวจ X ก่อน commit เสมอ") — คำตอบ
กำหนดพาธที่บันทึก:

| ประเภท | เฉพาะโปรเจกต์ | ทั่วเครื่อง (ทุกโปรเจกต์) |
| --- | --- | --- |
| สกิล | `.claude/skills/<ชื่อ>/SKILL.md` | `~/.claude/skills/<ชื่อ>/SKILL.md` |
| คำสั่ง | `.claude/commands/<ชื่อ>.md` | `~/.claude/commands/<ชื่อ>.md` |
| เอเจนต์ | `.claude/agents/<ชื่อ>.md` | `~/.claude/agents/<ชื่อ>.md` |
| กฎ (rules) | เพิ่มในไฟล์อ้างอิงของโปรเจกต์ (ดูด้านล่าง) | ไฟล์อ้างอิงส่วนตัวของผู้ใช้ (ถ้ามี) |

**สำคัญ:** ต้องอยู่ตรงชั้นนี้พอดี ห้ามสร้างโฟลเดอร์ย่อยเพิ่ม (เช่น
`.claude/skills/learned/<ชื่อ>/SKILL.md`) — Claude Code จะมองไม่เห็นและไม่โหลด
สกิล/คำสั่ง/เอเจนต์ที่ซ้อนโฟลเดอร์เกินระดับนี้เลย ถ้าซ้อนเกิน ไฟล์จะกลายเป็น
เอกสารเปล่าที่ไม่มีผลอะไร

## รูปแบบไฟล์ตามประเภท

### สกิล / คำสั่ง / เอเจนต์
ต้องใช้ **รูปแบบจริงของ Claude Code** เท่านั้น (มี YAML frontmatter `name`,
`description` ที่ถูกต้อง ตามแบบสกิลอื่นๆ ที่มีอยู่แล้วในโปรเจกต์นี้ เช่น
`.claude/skills/setup-pre-commit/SKILL.md`) ไม่ใช่ template อิสระ — ไฟล์ที่ไม่มี
frontmatter ที่ถูกต้องจะใช้งานไม่ได้จริง ให้ใส่บริบทที่มาไว้เป็นคอมเมนต์สั้นๆ
หรือย่อหน้าแรกของเนื้อหา ไม่ใช่แทนที่เนื้อหาที่ใช้งานได้จริง

### Hooks
**ห้ามบันทึกเป็นไฟล์ `hooks.json` เดี่ยวๆ — ไฟล์แบบนั้นไม่มีผลอะไรเลย** hook จะ
ทำงานได้ก็ต่อเมื่อถูก merge เข้าไปใน `settings.json` (`.claude/settings.json`
สำหรับโปรเจกต์ หรือ `~/.claude/settings.json` สำหรับทั่วเครื่อง) ใต้ key
`"hooks"` เท่านั้น เมื่อพบรูปแบบที่ควรเป็น hook ให้เรียกใช้สกิล `update-config`
(สกิลที่มีอยู่ในสภาพแวดล้อม Claude Code นี้ — ไม่ใช่ไฟล์เฉพาะของโปรเจกต์นี้)
เพื่อแก้ `settings.json` โดยตรง แทนการเขียนไฟล์ JSON แยกต่างหาก

### กฎ (Rules)
ใช้ template นี้ได้ (เอกสารอ้างอิงล้วนๆ ไม่มีกลไก auto-load ของ Claude Code
รองรับ — ถ้าเป็นกฎสั้นและสำคัญมากสำหรับโปรเจกต์นี้ ให้พิจารณาใส่ลงใน `CLAUDE.md`
โดยตรงแทน เพราะ `CLAUDE.md` ถูกโหลดทุกเซสชันอยู่แล้ว):

```markdown
# [ชื่อเรียกที่สื่อความหมายชัดเจน]

**บันทึกเมื่อ:** [วันที่]
**บริบทการใช้งาน:** [อธิบายสั้นๆ ว่ามีประโยชน์ในสถานการณ์ใด]

## ปัญหา
[ระบุปัญหาที่ต้องการแก้ไข — ให้ชัดเจนและเจาะจง]

## วิธีแก้ไข
[อธิบายรูปแบบการทำงาน เทคนิค หรือวิธีแก้ไขที่ได้ผล]

## ตัวอย่างการใช้งาน
[แสดงตัวอย่างโค้ดหรือขั้นตอนการปฏิบัติ หากเกี่ยวข้องและเป็นประโยชน์]

## เงื่อนไขการนำไปใช้
[ระบุสถานการณ์ที่ควรเรียกใช้เนื้อหานี้]
```

## ขั้นตอนการปฏิบัติ

1. ทบทวนบทสนทนา หาสิ่งที่ควรบันทึกตามหัวข้อ "สิ่งที่ควรนำมาบันทึก"
2. เลือกสิ่งที่มีคุณค่าที่สุดและนำกลับมาใช้ซ้ำได้จริง — ไม่ต้องบันทึกทุกอย่างที่เจอ
3. ระบุประเภท (สกิล/คำสั่ง/เอเจนต์/hook/กฎ) และ **ถามผู้ใช้ว่าเฉพาะโปรเจกต์หรือ
   ทั่วเครื่อง** ตามตารางด้านบน
4. จัดทำเนื้อหาตามรูปแบบที่ถูกต้องของประเภทนั้น (ดู "รูปแบบไฟล์ตามประเภท")
5. แสดงเนื้อหาที่จะบันทึกให้ผู้ใช้ยืนยันก่อนเขียนไฟล์จริงทุกครั้ง
6. บันทึกไฟล์ตรงตามพาธในตาราง (ไม่ซ้อนโฟลเดอร์เพิ่ม) — ถ้าเป็น hook ให้ไปแก้
   `settings.json` ผ่านสกิล `update-config` แทนการสร้างไฟล์ใหม่

## ข้อควรทราบ

- อย่าบันทึกวิธีแก้ปัญหาที่เกิดครั้งเดียวและง่ายเกินไป (พิมพ์ผิด, syntax error
  พื้นฐาน) เว้นแต่เกิดซ้ำเป็นรูปแบบตามเกณฑ์ข้อ 2
- เน้นบันทึกสิ่งที่ประหยัดเวลา ลดความผิดพลาด หรือทำให้งานครั้งต่อไปเป็นมาตรฐานได้จริง
- แต่ละไฟล์ครอบคลุมรูปแบบการทำงานหลักเดียวเท่านั้น อย่ารวมหลายเรื่องในไฟล์เดียว
- ก่อนสร้างสกิล/คำสั่ง/เอเจนต์ใหม่ ให้เช็คก่อนว่ามีของเดิมที่ควรอัปเดตแทนการสร้างซ้ำหรือไม่
